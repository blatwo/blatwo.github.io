                CREATE POLICY - 为表定义一个新的行级安全策略

                语法

                xxxxxxxxxx
                6
                1
                CREATE POLICY name ON table_name
                2
                    [ AS { PERMISSIVE | RESTRICTIVE } ]
                3
                    [ FOR { ALL | SELECT | INSERT | UPDATE | DELETE } ]
                4
                    [ TO { role_name | PUBLIC | CURRENT_ROLE | CURRENT_USER | SESSION_USER } [, ...] ]
                5
                    [ USING ( using_expression ) ]
                6
                    [ WITH CHECK ( check_expression ) ]

                描述

                CREATE POLICY命令为表定义一个新的行级安全策略。注意，必须在表上启用行级安全性（使用ALTER TABLE ... ENABLE ROW LEVEL SECURITY）才能应用创建的策略。

                策略授予选择、插入、更新或删除与相关策略表达式匹配的行的权限。现有表行将根据在USING中指定的表达式进行检查，而通过INSERT或UPDATE创建的新行将根据在WITH CHECK中指定的表达式进行检查。当USING表达式对于给定行返回true时，该行对用户可见；如果返回false或null，则该行对用户不可见。当WITH CHECK表达式对于行返回true时，该行将被插入或更新；如果返回false或null，则会发生错误。

                对于INSERT和UPDATE语句，WITH CHECK表达式在触发BEFORE触发器之后执行，并且在实际进行数据修改之前执行。因此，BEFORE ROW触发器可以修改要插入的数据，影响安全策略检查的结果。WITH CHECK表达式在任何其他约束之前执行。

                策略名称与表相关。因此，一个策略名称可以用于多个不同的表，并为每个表定义适合该表的策略。

                策略可以应用于特定的命令或特定的角色。新创建的策略的默认设置是对所有命令和角色应用，除非另有指定。一个命令可以应用多个策略；有关更多详细信息，请参阅下面的内容。Table 281总结了不同类型的策略如何应用于特定的命令。

                对于可以具有USING和WITH CHECK表达式（ALL和UPDATE）的策略，如果未定义WITH CHECK表达式，则USING表达式将用于确定哪些行是可见的（正常的USING情况），以及哪些新行允许添加（WITH CHECK情况）。

                如果表启用了行级安全性，但不存在适用的策略，则假定存在一个“默认拒绝”策略，因此不会显示或更新任何行。

                参数

                • name

                  要创建的策略的名称。该名称必须与表的任何其他策略的名称不同。

                • table_name

                  策略适用的表的名称（可选地包括模式限定符）。

                • PERMISSIVE

                  指定要创建的策略为宽容策略。对于给定查询适用的所有宽容策略将使用布尔“OR”运算符组合在一起。通过创建宽容策略，管理员可以添加可访问的记录。默认情况下，策略是宽容的。

                • RESTRICTIVE

                  指定要创建的策略为限制策略。对于给定查询适用的所有限制策略将使用布尔“AND”运算符组合在一起。通过创建限制策略，管理员可以减少可访问的记录，因为必须通过所有限制策略来访问每个记录。请注意，必须至少有一个宽容策略来授予对记录的访问权限，然后才能有用地使用限制策略来减少

                该访问。如果只存在限制性策略，则无法访问任何记录。当存在宽容和限制性策略的混合时，只有在通过至少一个宽容策略的同时通过所有限制策略时，才能访问记录。

                • command

                  应用策略的命令。有效选项为ALL、SELECT、INSERT、UPDATE和DELETE。ALL是默认值。有关如何应用这些选项的详细信息，请参见下面的内容。

                • role_name

                  要应用策略的角色。默认值为PUBLIC，将策略应用于所有角色。

                • using_expression

                  任何SQL条件表达式（返回布尔值）。条件表达式不能包含任何聚合或窗口函数。如果启用了行级安全性，该表达式将添加到引用该表的查询中。表达式返回true的行将可见。表达式返回false或null的行对用户不可见（在SELECT中），并且不能在UPDATE或DELETE中进行修改。这些行将被静默抑制；不会报告错误。

                • check_expression

                  任何SQL条件表达式（返回布尔值）。条件表达式不能包含任何聚合或窗口函数。如果启用了行级安全性，该表达式将用于针对表的INSERT和UPDATE查询。只允许表达式计算为true的行。如果表达式对于插入的任何记录或更新的任何记录计算为false或null，将抛出错误。请注意，check_expression是针对行的建议新内容进行评估，而不是原始内容。

                按命令类型的策略

                • ALL

                  使用ALL策略表示它将适用于所有命令，无论命令类型如何。如果存在ALL策略和更具体的策略，则将应用ALL策略和更具体的策略（或多个策略）。此外，如果只定义了USING表达式，则ALL策略将适用于查询的选择部分和修改部分，对于两种情况都使用USING表达式。例如，如果发出UPDATE语句，那么ALL策略将适用于UPDATE将能够选择要更新的行（应用USING表达式），以及适用于结果更新行，以检查它们是否允许添加到表中（应用WITH CHECK表达式，如果定义了，则使用USING表达式）。如果INSERT或UPDATE命令尝试添加不通过ALL策略的WITH CHECK表达式的行，则整个命令将中止。

                • SELECT

                  使用SELECT策略表示它将适用于SELECT查询和在定义策略的关系上需要SELECT权限的任何时候。结果是，只有通过SELECT策略的记录将在SELECT查询期间返回，需要SELECT权限的查询（例如UPDATE）也只能看到SELECT策略允许的记录。SELECT策略不能具有WITH CHECK表达式，因为它仅适用于从关系中检索记录的情况。

                • INSERT

                  使用INSERT策略表示它将适用于INSERT命令。不通过此策略的插入的行将导致策略违规错误，并中止整个INSERT命令。INSERT策略不能具有USING表达式，因为它仅适用于添加记录到关系的情况。请注意，带有ON CONFLICT DO UPDATE的INSERT仅对通过INSERT路径追加到关系中的行检查INSERT策略的WITH CHECK表达式。

                • UPDATE

                  使用UPDATE策略表示它将适用于UPDATE、SELECT FOR UPDATE和SELECT FOR SHARE命令，以及INSERT命令的辅助ON CONFLICT DO UPDATE子句。由于UPDATE涉及将现有记录替换为新的修改后的记录，因此UPDATE策略接受USING表达式和WITH CHECK表达式。USING表达式确定UPDATE命令将对其进行操作的记录，而WITH CHECK表达式定义了允许将修改后的行存储回关系中的行。任何更新值不通过WITH CHECK表达式的行将导致错误，并中止整个命令。如果只指定了USING子句，则该子句将用于USING和WITH CHECK情况。通常，UPDATE命令还需要从正在更新的关系中的列读取数据（例如，在WHERE子句或RETURNING子句中，或在SET子句的右侧表达式中）。在这种情况下，还需要对正在更新的关系进行SELECT权限，并且将应用适当的SELECT或ALL策略，除了UPDATE策略外，还必须通过SELECT或ALL策略授予用户对行的访问权限。当INSERT命令具有辅助的ON CONFLICT DO UPDATE子句时，如果采用UPDATE路径，则首先检查要更新的行是否通过任何UPDATE策略的USING表达式，然后检查新的更新行是否通过WITH CHECK表达式。然而，与独立的UPDATE命令不同，如果现有行未通过USING表达式，则会抛出错误（UPDATE路径将永远不会被静默避免）。

                • DELETE

                 

                使用DELETE策略表示它将适用于DELETE命令。只有通过此策略的行将被DELETE命令看到。可能会有通过SELECT可见但不可删除的行，如果它们不通过DELETE策略的USING表达式。在大多数情况下，DELETE命令还需要从要删除的关系中的列读取数据（例如，在WHERE子句或RETURNING子句中）。在这种情况下，还需要对关系进行SELECT权限，并且将应用适当的SELECT或ALL策略，除了DELETE策略外，还必须通过SELECT或ALL策略授予用户对行的访问权限。因此，用户必须通过SELECT或ALL策略访问要删除的行，除了通过DELETE或ALL策略授予权限删除行之外。DELETE策略不能具有WITH CHECK表达式，因为它仅适用于从关系中删除记录的情况，因此没有新行可以检查。

                多个策略的应用

                当不同命令类型的多个策略应用于同一命令时（例如，将SELECT策略和UPDATE策略应用于UPDATE命令），则用户必须同时具有两种类型的权限（例如，对行进行选择和更新的权限）。因此，一种类型的策略的表达式与另一种类型的策略的表达式使用AND运算符组合在一起。

                当同一命令类型的多个策略应用于同一命令时，必须至少有一个PERMISSIVE策略允许访问关系，并且所有RESTRICTIVE策略都必须通过。因此，所有PERMISSIVE策略表达式使用OR组合在一起，所有RESTRICTIVE策略表达式使用AND组合在一起，然后将结果使用AND组合。如果没有PERMISSIVE策略，则拒绝访问。

                注意事项

                必须是表的所有者才能为其创建或更改策略。

                虽然策略将应用于对数据库中的表的显式查询，但在系统执行内部引用完整性检查或验证约束时不会应用策略。这意味着有间接方法可以确定给定值是否存在。一个例子是尝试将重复的值插入作为主键或具有唯一约束的列。如果插入失败，则用户可以推断该值已经存在（此示例假设用户被授权插入他们无法看到的记录）。另一个例子是允许用户向引用另一个否则隐藏的表的表插入数据。用户可以通过将值插入引用表中来确定存在性，成功表示该值在引用表中存在。这些问题可以通过

                使用数据库事务进行并发控制来缓解。在某些情况下，使用外部模块或在数据库服务器外部运行的程序执行验证可能是有帮助的。

                在大多数情况下，只需指定USING表达式。只有在需要更复杂的访问控制时才需要WITH CHECK表达式。使用PERMISSIVE策略可以更轻松地为用户提供访问权限，而不需要为每个用户明确定义策略。

                对于性能和安全性的考虑，USING和WITH CHECK表达式不应包含聚合函数或窗口函数。

                要了解如何使用行级安全性和策略，请参阅Row-Level Security (RLS)。

                 

                行级安全（Row-Level Security，简称RLS）是一种在数据库中控制对行（数据记录）访问的安全策略。基于定义的策略，RLS 可以限制哪些用户能看到表中的哪些行，或者能对表中的哪些行进行修改。这可以有效地限制用户只能访问或修改他们自己的数据，即使这些数据存储在一个共享表中。

                例如，你可能有一个包含所有用户信息的表，你可以使用 RLS 确保用户只能看到和编辑他们自己的信息，而不能访问其他用户的信息。

                RLS 主要通过以下两种方式实现：

                1. 使用谓词：在定义安全策略时，可以为表的 SELECT, UPDATE, DELETE 操作定义一个条件（谓词）。当这个条件为真时，用户才能看到或者修改这些行。

                2. 使用表达式：可以为表的 INSERT, UPDATE 操作定义一个表达式。这个表达式需要在数据被插入或更新到表中前为真。如果表达式为假，则操作被阻止。

                这样，通过在数据库级别实现这些控制，可以进一步增强数据的安全性，尤其是在多用户和需要细粒度权限控制的环境中。

                RLS 功能在许多数据库系统中都有实现，包括 PostgreSQL，SQL Server 等。
